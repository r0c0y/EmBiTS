PageIndex Page-Aware Chunking & Document Navigation

To eliminate arbitrary text divisions across pages, ScribeLink integrates page-aware structural indexing inspired by the PageIndex and Haystack architectures:

  • Page-Bound Ingestion: During document ingestion, the file parser splits content page-by-page. For multi-page PDFs or OCR-scanned images, text is chunked within its respective page boundaries, and stored in the SQLite database with its explicit 1-indexed page_number.
  • Page-Aware RAG Citations: The search retrieval pipeline queries the page_number value along with raw text. This page number is injected directly into the LLM context prompt (e.g. * Excerpt (Chunk 4, Page 3): ...).
  • Explainable Provenance: The LLM is instructed to cite findings with precise page references (e.g., [1, Page 3]) instead of generic file tags. The citation lists in the UI render the exact page indicators for immediate verification by engineers.

Document Parsers & Ingestion Engine

The ingestion pipeline handles multiple formats natively on the local filesystem before splitting content into chunks:

  • PDF Documents: Parsed using pdfplumber and LiteParse to extract text and tables page-by-page. If the pages are scanned (no text layer) or have sparse text, the page image is dynamically rendered using pypdfium2 and sent to the OCR pipeline.
  • Word Files (.docx): Processed using python-docx to read paragraphs, headers, and bullet structures.
  • Excel Sheets (.xlsx): Loaded via openpyxl, converting active sheets to clean structured plain-text rows for indexing.
  • PowerPoint Slides (.pptx): Parsed with python-pptx, extracting text from shapes, slides, and notes.
  • Images (.png, .jpg): Loaded via pillow and processed through the RapidOCR ONNX Engine to pull OCR text.

Offline LiteParse Tesseract Configuration:

For fully air-gapped/offline installations utilizing LiteParse's built-in Tesseract engine, download the language files (e.g. eng.traineddata) and configure the environment variable:

export TESSDATA_PREFIX=/path/to/tessdata_directory

Coordinate Space Conversion: Bounding boxes are extracted in standard PDF points (1/72 inch). To display highlights on a rendered page image at 150 DPI, the system maps coordinates using a scaling multiplier: Scale = DPI / 72 = 150 / 72 ≈ 2.083.

High-Accuracy PP-OCRv5 Offline Model Upgrade

For processing low-resolution scans, hand-written minutes, or multilingual documents (such as Hindi or Latin text), the ingestion engine supports drop-in high-accuracy PP-OCRv5 model weights in ONNX format:

  1. Create a folder at hosted/models/ocr/.
  2. Download the PP-OCRv5 detection model (det.onnx) and recognition model (rec.onnx + dict.txt) from monkt/paddleocr-onnx on Hugging Face.
  3. Place the files inside the folder: 
    hosted/models/ocr/det.onnx
hosted/models/ocr/rec.onnx
hosted/models/ocr/dict.txt
  4. The engine will automatically detect these files on startup and load the high-accuracy models instead of the default standard ones.

Hybrid Retrieval & RRF Sequence

The search engine uses a parallel hybrid retrieval strategy combining sparse (BM25) and dense semantic matches, resolved via Reciprocal Rank Fusion (RRF):

sequenceDiagram autonumber User->>Search Engine: Input Search Query ("Why did Lot 2 yield drop?") par Parallel Path 1: Classical Search Search Engine->>SQLite FTS5: Match query tokens (BM25 ranker) SQLite FTS5-->>Search Engine: Return Ranked Keyword Chunks and Parallel Path 2: Semantic Search Search Engine->>Ollama (Embeddings): Request Query Vector (embeddinggemma) Ollama (Embeddings)-->>Search Engine: Return 768-dim Query Vector Search Engine->>SQLite Database: Load Packed float32 BLOB vectors Search Engine->>Vector Store: Compute Cosine Similarity (NumPy) Vector Store-->>Search Engine: Return Ranked Semantic Chunks end Search Engine->>RRF Merger: Merge Rank lists A & B using RRF Formula RRF Merger-->>Search Engine: Select Top Chunks (Normalized confidence 0.0 - 0.99) Search Engine->>Ollama (LLM): Call /api/chat with Chunks + Query (gemma3:1b) Ollama (LLM)-->>Search Engine: Generate QA Response (concise/elaborate XML tags) Search Engine-->>User: Display AI Answer & Cytoscape Traceability Nodes

Mathematical Fusion (Reciprocal Rank Fusion):

Both retrieval systems output an ordered list of matched chunks. To merge them fairly without scale mismatch, we use the Reciprocal Rank Fusion score for each document chunk d:

Score(d) = 1 / (60 + Rank_Keyword(d)) + 1 / (60 + Rank_Vector(d))

Where Rank(d) is the 1-based position of chunk d in the respective results list (defaulting to infinity if the chunk is not present in that list). A constant factor of 60 is used to mitigate the influence of outlier high ranks. The final RRF scores are normalized to a 0.0 - 0.99 range to display as relevance confidence levels in the browser.

Configuration & Customization

Guidelines to modify system components, swap databases, and adjust LLM parameters.

1. Swapping SQLite with PostgreSQL (PGVector)

To upgrade to a production PostgreSQL database, modify database.py to use a PostgreSQL connection string instead of local SQLite:

# In database.py
import psycopg2

def get_db_connection():
    # Replace SQLite connection with PostgreSQL
    return psycopg2.connect("postgresql://user:password@localhost:5432/scribelink")

To support vectors, enable the PGVector extension in PostgreSQL:

CREATE EXTENSION IF NOT EXISTS vector;
CREATE TABLE chunk_embeddings (
    chunk_id INT REFERENCES document_chunks(id) ON DELETE CASCADE,
    vector vector(768)
);

2. Adjusting LLM & Embedding Models

Modify config.py to target different local Ollama models (e.g., Llama 3, Qwen 2.5) or adjust API endpoints:

# In config.py
OLLAMA_BASE_URL = "http://localhost:11434"
LLM_MODEL = "gemma3:1b"           # Generation model
EMBEDDING_MODEL = "embeddinggemma:latest" # Embedding model
