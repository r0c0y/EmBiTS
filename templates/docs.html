Running the Server

Start with python main.py. By default the server runs with auto-reload disabled. To enable hot-reload during development:

SCL_RELOAD=true python main.py

Opens at http://localhost:8000. First user to sign up gets admin role.

Core Data Flow

  1. Upload: File received by FastAPI → SHA-256 hash + duplicate check → DB row (status: pending) → temp file → background task returns immediately
  2. Background Processing: OCR/extraction → page-aware form-aware chunking (800 char sliding window) → FTS5 index + vector embeddings → RAPTOR hierarchical summaries → SHA-256 audit log
  3. Search: Meta-query detection (bypass for "list all documents") → parallel FTS5 BM25 + vector cosine similarity → RRF merge (k=60) → token title-boost + length normalization → query-aware context ordering → top 12 chunks, max 3 per doc → RAPTOR summaries injected per-source → LLM with <concise>/<elaborate> output → parameter conflict scan → graph with shared-query-topic edges
  4. Audit: Every action (SIGNUP, SIGNIN, QUERY, UPLOAD) logged to SHA-256 chained ledger for tamper-evident compliance

RRF Score: Score(d) = 1/(60 + rank_k(d)) + 1/(60 + rank_v(d))

Title boost: +5 per query term matched in document title

Length normalization: 1.0 / max(1.0, (doc_len / 1000) ^ 0.3)

Ingestion Pipeline

End-to-end flow from file upload to fully indexed document, including format-specific parsers, form-aware chunking, and background processing.

Upload Handler — main.py:upload_document()

Request Validation

  • Validates file extension against allowed set (.txt .pdf .docx .xlsx .xls .pptx .png .jpg .jpeg .tiff .bmp .csv .md .json)
  • Computes SHA-256 hash of raw bytes
  • Duplicate check: same content_hash or same file_path + same project_id → returns 409 with duplicate status
  • Filename sanitization: removes non-alphanumeric chars (except space, underscore, hyphen), truncates to 200 chars

DB Insertion

Inserts into meetings table with status = "pending". All 20 columns:

Background Processing — _process_upload_background()

  1. Copy original file to static/originals/{doc_id}_{filename}
  2. Call ingestion.ingest(file_path, filename) → returns ExtractionResult dict
  3. Generate lot_id as LOT-{YYYY-MM}-{last4chars}
  4. Update meeting record with transcript, date, OCR metadata
  5. Store OCR sidecar via ocr_storage.store_ocr_result()
  6. Insert page-aware chunks into chunks table (per-page chunking for OCR docs, single-pass for native text)
  7. Log audit entry (UPLOAD) with chunk count and engine name
  8. Call auto_edges() for lineage generation
  9. Trigger background summarization + embedding building
  10. Set status = "completed" (or "failed" on error with message in transcript_text)

ingestion.ingest() — File Parsing Router

ingestion.py routes files by extension. Receives a file path string for clean temp-file management.

Image & PDF Formats

For .pdf .png .jpg .jpeg .tiff .bmp: calls OCRManager.process_with_fallback(file_path, ext) with the path string.

Form-Aware Chunking — chunk_text()

Sliding window with 800 character chunks, 200 character overlap. Before sliding-window splitting, the chunker detects form/template structure:

  • Scans for repeated Field Name: or Field Name : patterns
  • If ≥3 field patterns detected AND total text < 2400 chars, the entire document is kept as a single chunk
  • Otherwise: splits on paragraph boundaries (double newlines) → long paragraphs split on sentence boundaries → 800-char sliding window with 200-char overlap

Date Detection — detect_date()

Multi-format regex scanning in priority order:

PatternExample
YYYY-MM-DD2025-03-15
DD/MM/YYYY or DD-MM-YYYY15/03/2025 (tries multiple orderings)
Month DD, YYYYMar 15, 2025
DD Month YYYY15 March 2025

First match wins. Returns YYYY-MM-DD format.

Engine Details

1. LiteParseEngine (Priority 110)

Uses liteparse library for layout-aware PDF parsing with reading order detection. Handles multi-column layouts, tables, and embedded figures. Falls back per-page via is_text_garbled() → attempts native PyMuPDF text → then Tesseract OCR.

2. PyMuPDFEngine (Priority 100)

Native PDF text extraction via fitz (PyMuPDF). Provides low-level text blocks with bounding box coordinates. Processes pages in parallel using ThreadPoolExecutor (up to 8 workers).

3. TesseractEngine (Priority 90)

Uses pytesseract (Tesseract OCR engine). Applied as the final fallback for scanned images and garbled pages. Includes pre-processing via preprocess.py pipeline. Requires Tesseract system installation.

Garbled Text Detection — is_text_garbled()

Multi-heuristic check at ocr_engine.py:

  1. Empty text: If page text has fewer than 10 characters → garbled
  2. Unicode replacement character: If \ufffd present → garbled
  3. Short average word length: If text < 200 chars and avg word length < 2.0 → garbled
  4. Alpha character ratio: Lines where alpha-char ratio < 0.4 → garbled line
  5. Allowed character ratio: Lines where allowed-char ratio < 0.75 → garbled line
  6. Threshold: If > 20% of lines are garbled → entire page is garbled

Triggers per-page intra-engine fallback: e.g., LiteParse processes pages 1-2, page 3 is garbled → page 3 routed to PyMuPDF → if still garbled → Tesseract OCR.

Shadow Text Cleaning — clean_shadow_text()

Post-processing function that cleans double-rendered shadow characters (common in scanned PDFs where bold/outline text renders twice at slightly offset positions):

  • Identifies words where ≥75% of characters are duplicated (e.g., GRROOUUPP → GROUP)
  • Compresses each repeated character run by half
  • Applied after OCR extraction before chunking

Image Preprocessing — preprocess.py

Applied before Tesseract OCR to improve accuracy on noisy/scanned input:

  1. Alpha Channel Blending — blends RGBA transparency with white background
  2. Grayscale conversion — cv2.cvtColor(img, cv2.COLOR_BGR2GRAY)
  3. Deskew — detects rotation via cv2.minAreaRect, rotates to correct
  4. Bilateral Filtering — smooths noise while preserving sharp font edge contours
  5. Adaptive Binarization — converts to high-contrast binary pixel states using Gaussian threshold
  6. Median Blur — removes salt-and-pepper noise

Sidecar Storage — ocr_storage.py

After OCR processing, results are persisted to:

FileContentPurpose
storage/ocr/{doc_id}.jsonPer-page blocks with bounding boxes, markdown, metadata, engine name, sorted by reading order (y-coordinate with line tolerance then x-coordinate)API consumption, viewer rendering, block-level retrieval

Search & Retrieval

RRF hybrid search system combining BM25 keyword matching with dense vector semantic search, title-boost and length normalization, per-document chunk caps, query-aware context ordering, RAPTOR summaries, conflict detection, and graph edge labels.

1. Meta-Query Detection

search_engine.py:is_meta_query() — Trigger phrases like "list all documents", "all files", "project list", "list meetings" bypass hybrid search entirely and return a direct SQL listing formatted as LLM summary with metadata.

2. FTS5 Keyword Search — keyword_search()

  • Query tokenized via tokenize(): stopwords removed, porter stemmer applied via FTS5
  • SQLite chunks_fts MATCH with bm25() ranking
  • Project/document/date filters applied in WHERE clause
  • Returns top 30 results

Title Fallback

If FTS returns zero results but query terms exist, falls back to matching document titles directly:

  • Scans all documents (respecting filters)
  • If ≥2 query terms match the title (or ≥1 with ≤3 query terms), creates a synthetic chunk entry with score = title_matches * 10.0
  • This ensures documents are findable even when FTS5 has no verbatim matches

Title Boost & Length Normalization

Every FTS5 result is post-processed:

  • Title boost: +5 × title_matches where title_matches = number of query terms found in the document title
  • Length normalization: 1.0 / max(1.0, (doc_len / 1000) ^ 0.3) prevents huge documents from dominating BM25 rankings
  • score = bm25_score × length_norm + title_bonus

3. Vector Semantic Search

  • Query embedded via Ollama embeddinggemma:latest → 768-dim float32 vector
  • All candidate chunks loaded from chunk_embeddings BLOB columns (raw float32 pack, 3072 bytes)
  • Cosine similarity computed in Python (NumPy): cosine_similarity(q, v) = dot(q, v) / (norm(q) × norm(v))
  • Returns top 30 results by similarity score

4. RRF Merger

Score(d) = 1/(60 + rank_k(d)) + 1/(60 + rank_v(d))

  • Constant 60 mitigates outlier high ranks
  • Scores summed across FTS5 + vector result sets
  • Results sorted by combined score, top 15 returned
  • Final confidence normalized to 0.0–0.99: min(0.99, rrf_score × 30)

5. Context Building

Per-Document Chunk Limit

Chunks are first sorted by _qscore (hybrid of RRF score + query token overlap ratio):

chunk._qscore = score * 0.3 + (query_terms ∩ chunk_terms) / max(query_terms, 1) * 0.7

Top 12 chunks are grouped by document, with a maximum of 3 chunks per document. Within each group, chunks are sorted by chunk_index (chronological page order).

Query-Aware Document Ordering

Document groups are sorted by _max_qscore (descending), putting the most query-relevant document first — NOT chronological order. This ensures the LLM sees the most relevant source material first regardless of document date.

RAPTOR Summaries Per-Source

Each document's Level-1 hierarchical summary is fetched from hierarchical_summaries and placed before its chunk excerpts in the prompt. The prompt structure for each source:

Source [N] (ID: DOC-XXXXXXXX):
Document Title: {title}
Document Date: {date}
  * Summary: {L1 hierarchical summary}
  * Excerpt (Chunk N, Page X): {chunk text}
  * Excerpt (Chunk N+1, Page Y): {chunk text}

6. LLM Generation

  • System prompt: Instructs LLM to use [SourceNumber] citations matching source numbers in context, emit no raw doc IDs or HTML, output in clean markdown only
  • Output format: Wrapped in <concise> (detailed synthesis) and <elaborate> (deep-dive chronological analysis) XML tags
  • Post-processing: If LLM omits XML tags, they are reconstructed: first 3 bullet points → <concise>, full text → <elaborate>
  • Primary: Ollama gemma3:1b via /api/chat with 120s timeout
  • Fallback: Groq llama-3.1-8b-instant (optional, controlled by USE_GROQ_FALLBACK env var)
  • Translation: English ↔ Hindi via Ollama, cached per text string, Devanagari script emphasis with natural grammar

7. Conflict Detection — conflict.py

Scans retrieved document full text for semiconductor parameter patterns and flags contradictions:

M1 Min SpacingMetal\s*1\s*(?:Minimum\s+)?Spacing
M1 ThicknessMetal\s*1\s*Thickness (Angstrom)
Gate Oxide ThicknessGate\s*Oxide\s*Thickness (Angstrom)

When two documents cite different values for the same parameter, a conflict entry is generated with both values, source documents, and context snippets (30 chars before/after match).

8. Graph Edge Labels

search_helpers.py:get_context_graph() generates edges between documents with a label field:

  • Documents that share query terms → edge label shows comma-separated shared terms (truncated to 20 chars)
  • Documents in the same project → edge label shows project ({name})
  • Documents in the same lot → edge label shows lot ({lot-id})
  • Frontend graph.js renders labels inline on edges with CSS auto-rotation

Audit Ledger

Cryptographically chained, tamper-evident audit log using SHA-256 block hashing.

Chain Structure

graph LR Genesis[Genesis Block
parent_hash: "GENESIS"] --> Block1[Block 1
parent_hash: hash0
current_hash: hash1] Block1 --> Block2[Block 2
parent_hash: hash1
current_hash: hash2] Block2 --> Block3[Block N
parent_hash: hashN-1
current_hash: hashN]

Hash Calculation

import hashlib
raw = (timestamp + username + department + action_type + details + parent_hash).encode("utf-8")
current_hash = hashlib.sha256(raw).hexdigest()

The parent_hash of each new entry is the previous entry's current_hash: SELECT current_hash FROM audit_logs ORDER BY id DESC LIMIT 1.

The first entry uses "GENESIS" as its parent_hash.

Verification — verify_audit_ledger()

Iterates all entries in ID-ascending order:

  1. Check that parent_hash of entry N matches current_hash of entry N-1
  2. Recalculate SHA-256(timestamp + username + ... + parent_hash) and confirm it matches stored current_hash
  3. Return {valid: true, count: N} or {valid: false, failed_id: N}

API: POST /api/audit_logs/verify returns verification result.
